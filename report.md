# Lab 16 Benchmark Report: Reflexion vs ReAct Agent Architecture

## 1. Overview & Core Mechanism
This report evaluates the implementation of a **Reflexion Agent** architecture designed to solve complex multi-hop reasoning tasks. Unlike traditional single-pass models (ReAct), the Reflexion framework introduces a self-correcting loop that enables the model to learn from its own mistakes in real-time.

### 1.1 Architecture & Flow
The system operates through a closed-loop interaction between three distinct functional roles:

```mermaid
graph TD
    Start((Input Question)) --> Actor
    Actor[<b>Actor (Reasoning)</b><br/>Generates initial answer based on context] --> Judge
    Judge[<b>Evaluator (Validation)</b><br/>Verifies answer against Ground Truth]
    Judge -- Correct --> Success((Final Success))
    Judge -- Incorrect --> Reflector
    Reflector[<b>Reflector (Analysis)</b><br/>Identifies failure logic & suggests correction] --> Actor
    
    subgraph "The Reflexion Loop"
    Actor
    Judge
    Reflector
    end
```

### 1.2 Agent Responsibilities
- **Actor (Reasoning)**: Processes the provided context chunks and the question. It is designed to be highly sensitive to previous reflection strategies, ensuring that it does not repeat the same reasoning path if it was previously flagged as incorrect.
- **Evaluator (Validation)**: Performs an objective assessment of the candidate answer. It ensures that the model stays grounded in the gold standard truth provided in the dataset.
- **Reflector (Meta-Cognition)**: This is the brain of the recovery process. When a failure occurs, it analyzes the gap between the predicted answer and the required evidence, generating a "Next Strategy" that guides the Actor towards the correct reasoning hop.

---

## 2. Experimental Methodology
- **Core Strategy**: Implementation of a recursive loop with **Adaptive Max Attempts**.
- **Difficulty-Aware Logic**: The system dynamically scales its reasoning depth based on the perceived complexity of the task:
    - **Hard Questions**: Allocated up to 5 attempts to allow for deep multi-hop resolution.
    - **Medium Questions**: Allocated 3 attempts.
    - **Easy Questions**: Restricted to 1 attempt to maximize efficiency.
- **Evaluation Metric**: Exact Match (EM) score calculated after normalization.

## 4. Key Mechanisms Implemented
Beyond the standard architecture, several advanced reasoning extensions were integrated:

- **Adaptive Reasoning Depth**: Adjusting the number of allowed reflections based on the difficulty level of the question.
- **Reflection Memory Stream**: Maintaining a history of failed reasoning paths to prevent the model from entering infinite loops or repeating hallucinations.
- **Structured Self-Correction**: Ensuring that the Reflector provides actionable, step-by-step instructions (Next Strategy) rather than just identifying errors.
- **Difficulty-Based Performance Tracking**: A granular breakdown of success rates to identify specific types of reasoning failures.

---

## 5. Discussion & Conclusion
1. **Model Recovery**: The Reflexion mechanism proved highly effective at correcting "incomplete hop" errors, where the model initially identifies the correct intermediate entity but fails to reach the final answer.
2. **Resource Tradeoff**: While Reflexion provides a safety net for reasoning, it comes with a significant increase in computational latency. This suggests that the mechanism should be used selectively for complex, high-uncertainty tasks.
3. **Loop Dynamics**: The success of the second and third attempts heavily depends on the quality of the "Lesson" generated by the Reflector. High-quality reflections lead to near-instant correction, while poor reflections can lead to "reflection overfit."

In conclusion, the **Reflexion architecture** successfully provides a robust framework for autonomous self-correction, significantly enhancing the reliability of the model on complex multi-hop reasoning tasks.
